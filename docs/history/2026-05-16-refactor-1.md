## Why This Refactor

The codebase had grown to a point where four view files exceeded 900 lines, the database schema was a 931-line monolith, schema migrations had no history, and there was zero test coverage for the highest-risk domain logic. None of these were blocking features today — but all of them would become painful the moment a second developer joined, a schema migration went wrong, or a scoring regression went undetected. The goal was to fix the structural debt before adding any new features.

---

## Sprint 1 — Safety & Data Integrity

**Problem:** `drizzle-kit push` applied schema changes directly to the database with no migration history and no rollback path. Renaming a column would silently drop data. Hardcoded date constants in `plan-phases.ts` duplicated data already in the database, meaning the two sources would silently diverge whenever sprint dates were updated through the UI.

**What changed:**
- Switched `db:push` to `db:generate` + `db:migrate` workflow. Migration files now live in `apps/web/src/lib/db/migrations/` and are committed to git before being applied.
- Renamed `db:push` to `db:push:dev` — clearly marked unsafe for production.
- Updated `plan-phases.ts` to accept runtime `planStart`/`planEnd` parameters. The `/learning-plan` page now derives phase logic from database sprint data, removing the static date constants as the source of truth.
- Added `userId` filter to the projects query so `getActiveProject()` is properly user-scoped.

**Files:** `apps/web/package.json`, `drizzle.config.ts`, `lib/ielts/plan-phases.ts`, `lib/db/projects.ts`, `app/(dashboard)/learning-plan/`

## Sprint 2 — Code Quality & Reliability

**Problem:** The 931-line `schema.ts` covered 50+ tables across unrelated domains in one file — every new feature caused merge conflicts, and the implicit ordering of table definitions at the top vs. relations at the bottom made the file hard to navigate. No shared result type existed for server actions, so callers had no structured signal when a DB call failed.

**What changed:**
- Split `lib/db/schema.ts` into six domain files under `lib/db/schema/`:
  - `auth.ts` — users, sessions, feedback results
  - `content.ts` — vocabulary, collocations, reading, listening, writing topics
  - `practice.ts` — mock exams, drill results, wrong decision logs, connected speech
  - `projects.ts` — projects, sprints, tickets, comments, completions
  - `user-data.ts` — user vocabulary, collocations, sentence practice, SRS, word pairs
  - `admin.ts` — page configs, user usage, campaign configs, feedback, partner inquiries
  - `index.ts` re-exports all tables — all existing `import { X } from '@/lib/db/schema'` imports continue to work unchanged.
- Added `lib/actions.ts` with a shared `ActionResult<T>` type (`{ ok: true; data: T } | { ok: false; error: string }`) for new server actions going forward.
- Corrected `revalidatePath` scope on two admin actions: `activateUserAction` and `updateUserTierAction` now revalidate `/admin/users` instead of `('/', 'layout')`.
- Removed `console.log` calls from app and lib code (kept intentional seed progress output).

**Files:** `lib/db/schema/` (6 new files + index), `lib/actions.ts`, `app/actions/admin.ts`

## Sprint 3 — Large File Splitting

**Problem:** Four view files had grown beyond 900 lines each, mixing data-fetching logic, local state, event handlers, and render output in a single file. The 1,309-line `essay-builder-view.tsx` was the worst offender — a single file contained utility functions, three distinct panel components, a history tab, and a full AI analysis workflow.

**Rule applied:** Extract sub-components into a `components/` subfolder next to the page. A file over 500 lines is a signal it is doing too much.

**What changed:**

| File (before) | Lines | Extracted to |
|---|---|---|
| `essay-builder/essay-builder-view.tsx` | 1,309 | `utils.tsx`, `components/version-row.tsx`, `components/selection-panel.tsx`, `components/history-card.tsx` |
| `wrong-decisions/wrong-decisions-view.tsx` | 1,078 | `components/constants.ts`, `components/stats-bar.tsx`, `components/entry-form.tsx`, `components/log-card.tsx` |
| `writing/writing-task.tsx` | 985 | `components/topic-card.tsx`, `components/audit-panel.tsx`, `components/gap-panel.tsx`, `components/sample-panel.tsx` |
| `speaking/drill/drill-view.tsx` | 929 | `utils.ts`, `components/mistake-card.tsx` |

After extraction, the orchestrator files are 241–766 lines — each responsible only for composing its sub-components and managing top-level state.

**Key pattern:** Types used by both orchestrator and sub-component live in the sub-component file and are re-imported by the orchestrator (e.g. `GapCriterion` lives in `gap-panel.tsx`). Client-safe constants (SKILLS, SKILL_LABELS, etc.) were moved to `components/constants.ts` to avoid the `Module not found: fs` error when a `'use client'` component transitively imports from `lib/db/*`.

## Sprint 4 — Testing & Observability

**Problem:** Zero test infrastructure. The highest-risk code — phase detection logic, filler detection, examiner prompt contracts — had no coverage. Regressions in scoring or phase calculation would be invisible until a user noticed.

**What changed:**
- Installed `vitest` + `@vitest/ui` at workspace root.
- Added `test` and `test:ui` scripts to `apps/web/package.json`.
- Created `apps/web/vitest.config.ts` with `src/**/*.test.ts` include pattern (excludes Playwright e2e files).
- Wrote 41 unit tests across three files:

| Test file | Tests | What it covers |
|---|---|---|
| `lib/ielts/feedback/filler-detector.test.ts` | 12 | `detectFillers`: empty input, single/repeated fillers, multi-filler detection, case-insensitivity, sort order, word boundary guards. `totalFillerCount`: sum and edge cases. |
| `lib/ielts/plan-phases.test.ts` | 13 | `getCurrentPhaseStatus`: before/active/complete states, correct phase id, day-into-phase calculation, overall percentage, custom plan bounds. `daysUntil`: positive, zero, negative. Phase array: chronological order, phase structural integrity. |
| `lib/ielts/examiner/prompt.test.ts` | 16 | All four examiner prompt functions (Part 1 with/without topic, Part 2, Part 3, cue card generator) and `FEEDBACK_SYSTEM_PROMPT` constant: role identity, topic injection, no-feedback rules, JSON-only output contract. |

Sprint 4.2 (Neon serverless DB driver) was intentionally skipped — the project self-hosts on a persistent Docker server. The TCP driver (`postgres`) has no connection limit risk in this configuration.
